Clase 16 — Clases en JavaScript

1. ¿Qué son las clases?

Las clases fueron introducidas en ES6.

Su objetivo es ofrecer una sintaxis más clara para trabajar con objetos y prototipos.

Error muy común: pensar que las clases sustituyen a los prototipos.

Eso es falso.

Las clases son una capa de sintaxis sobre el sistema de prototipos que ya existía en JavaScript.

Por debajo, JavaScript sigue utilizando prototipos para compartir métodos y construir relaciones entre objetos.

Entender prototipos hace mucho más fácil entender clases.

2. Antes de class: funciones constructoras

Antes de existir la sintaxis class, JavaScript utilizaba funciones constructoras para crear objetos con una estructura común.

function Usuario(nombre, apellidos) {
  this.nombre = nombre;
  this.apellidos = apellidos;
}

const u1 = new Usuario("Bienve", "Sáez");
const u2 = new Usuario("Felipe", "Pérez");

Usuario es una función constructora.

u1 y u2 son objetos distintos.

Una función constructora actúa como una plantilla para crear objetos con propiedades similares.

Error típico: pensar que una función constructora es una clase tradicional como en Java o C#.

3. ¿Qué hace realmente new?

Cuando usamos new, JavaScript no se limita a ejecutar una función.

const u1 = new Usuario("Jonás", "Campos");

Internamente ocurren varios pasos.

Paso 1: JavaScript crea un objeto vacío.

{}

Paso 2: conecta ese objeto con el prototipo de la función constructora.

u1
 ↑
Usuario.prototype

Paso 3: ejecuta la función constructora usando ese objeto como this.

Usuario.call(u1, "Jonás", "Campos");

Paso 4: devuelve el objeto creado.

new crea el objeto.

new lo conecta con su prototipo.

new ejecuta la función constructora.

new devuelve el objeto terminado.

No memorices únicamente la sintaxis.

Lo importante es comprender el proceso que ocurre detrás.

4. Métodos dentro del constructor

Podemos crear métodos directamente dentro de la función constructora.

function Usuario(nombre, apellidos) {
  this.nombre = nombre;
  this.apellidos = apellidos;

  this.saludar = function () {
    console.log(`Hola, soy ${this.nombre} ${this.apellidos}`);
  };
}

Esto funciona correctamente.

Al usar this.saludar, cada objeto recibe una propiedad propia llamada saludar.

Ejemplo:

const u1 = new Usuario("Pepe", "García");
const u2 = new Usuario("Juan", "Pérez");

Tanto u1 como u2 tienen una propiedad propia llamada saludar.

Cada instancia crea una función nueva.

console.log(u1.saludar === u2.saludar);

Resultado:

false

Aunque el código sea idéntico, JavaScript crea dos objetos función distintos.

Las funciones son objetos.

Los objetos se comparan por referencia, no por contenido.

Si todos los objetos necesitan exactamente el mismo comportamiento, crear una función nueva para cada instancia es innecesario.

5. Métodos en el prototype

Si todos los objetos necesitan el mismo comportamiento, podemos almacenar el método en el prototipo.

function Usuario(nombre, apellidos) {
  this.nombre = nombre;
  this.apellidos = apellidos;
}

Usuario.prototype.saludar = function () {
  console.log(`Hola, soy ${this.nombre} ${this.apellidos}`);
};

Ahora saludar ya no vive dentro de cada objeto.

Existe una única función compartida.

Visualización mental:

u1
 ↑
Usuario.prototype
 └─ saludar()

u2
 ↑
Usuario.prototype
 └─ saludar()

Tanto u1 como u2 utilizan exactamente el mismo método.

console.log(u1.saludar === u2.saludar);

Resultado:

true

Ambos están apuntando al mismo objeto función.

Error típico: pensar que JavaScript copia el método dentro de cada objeto.

El método se comparte mediante la cadena de prototipos.

6. Convertimos la función constructora en clase

La sintaxis class permite escribir lo mismo de una forma más clara.

class Usuario {
  constructor(nombre, apellido) {
    this.nombre = nombre;
    this.apellido = apellido;
  }

  saludar() {
    console.log(`Hola, soy ${this.nombre} ${this.apellido}`);
  }
}

const u1 = new Usuario("Bienvenido", "Sáez");
u1.saludar();

constructor() es un método especial.

new crea el objeto y prepara todo el proceso.

constructor() inicializa las propiedades de ese objeto.

Los métodos definidos dentro de una clase se almacenan automáticamente en Usuario.prototype.

La sintaxis es más cómoda.

El mecanismo interno sigue siendo el mismo.

Las clases no sustituyen a los prototipos.

Las clases utilizan prototipos.

7. Propiedades privadas con #

JavaScript moderno permite crear propiedades privadas mediante el símbolo #.

class Usuario {
  #edad;

  constructor(nombre, apellido, edad) {
    this.nombre = nombre;
    this.apellido = apellido;
    this.#edad = edad;
  }

  saludar() {
    console.log(
      `Hola, soy ${this.nombre} ${this.apellido} y tengo ${this.#edad} años`
    );
  }
}

#edad es una propiedad privada.

Solo puede utilizarse desde dentro de la propia clase.

Esto produce un error:

const u1 = new Usuario("Pepe", "García", 30);

console.log(u1.#edad);

El código externo no puede acceder directamente a una propiedad privada.

“Privado” no significa secreto absoluto.

Significa que la propiedad es de uso interno de la clase.

Visualización mental:

Exterior
   ❌
#edad
   ✔
Clase Usuario

Esto permite proteger datos y controlar cómo se utilizan.

8. Métodos privados

Además de propiedades privadas, también podemos crear métodos privados.

class Usuario {
  salir() {
    this.#cerrarSesion();
  }

  #cerrarSesion() {
    console.log("He cerrado la sesión");
  }
}

const u1 = new Usuario();
u1.salir();

salir() es un método público.

#cerrarSesion() es un método privado.

Los métodos públicos pueden utilizar métodos privados internamente.

Esto produce un error:

u1.#cerrarSesion();

Los métodos privados no pueden ejecutarse desde fuera de la clase.

Visualización mental:

u1.salir()
     ↓
#cerrarSesion()
     ↓
"He cerrado la sesión"

Los métodos privados sirven para ocultar detalles internos de funcionamiento.

El usuario utiliza la interfaz pública sin conocer la implementación interna.

9. Métodos estáticos

Un método estático pertenece a la clase.

No pertenece a los objetos creados por la clase.

class Usuario {
  constructor(nombre, apellido) {
    this.nombre = nombre;
    this.apellido = apellido;
  }

  static crearInvitado() {
    return new Usuario("Invitado", "Anónimo");
  }
}
const invitado = Usuario.crearInvitado();

El método se ejecuta desde la propia clase.

Usuario.crearInvitado();

No se ejecuta desde una instancia.

const u1 = new Usuario("Jonás", "Campos");

u1.crearInvitado(); // Error

Un método estático pertenece a la clase, no a los objetos creados.

Uso típico:

Crear objetos especiales
Funciones auxiliares
Utilidades relacionadas con la clase

10. ¿Dónde viven los métodos estáticos?

Los métodos normales viven en el prototipo.

Usuario.prototype.saludar

Los métodos estáticos viven directamente en la clase.

Usuario.crearInvitado

Visualización mental:

Usuario
 └─ crearInvitado()

Usuario.prototype
 └─ saludar()

u1
 ↑
Usuario.prototype

Los métodos normales son accesibles desde las instancias.

Los métodos estáticos son accesibles desde la clase.

Error típico:

u1.crearInvitado();

Las instancias no tienen acceso a los métodos estáticos.

11. Getters

Un getter permite ejecutar lógica cuando leemos una propiedad.

class Usuario {
  #edad;

  get edad() {
    console.log("Ejecutando getter");
    return this.#edad;
  }
}
console.log(u1.edad);

Parece una propiedad normal.

Internamente se ejecuta una función.

El getter permite controlar cómo se obtiene un valor.

Error típico:

u1.edad();

Un getter no se usa como una función.

Se utiliza exactamente igual que una propiedad.

12. Setters

Un setter permite ejecutar lógica cuando asignamos un valor.

class Usuario {
  #edad;

  set edad(nuevaEdad) {
    if (nuevaEdad < 0) {
      throw Error("La edad no puede ser menor que 0");
    }

    this.#edad = nuevaEdad;
  }
}
u1.edad = 25;

Aunque parece una asignación normal, internamente se ejecuta el setter.

Los setters permiten validar datos antes de guardarlos.

Ejemplo:

u1.edad = -10;

Resultado:

Error: La edad no puede ser menor que 0

El setter evita que el objeto entre en un estado inválido.

13. Getters y setters juntos

Lo habitual es utilizar getter y setter para controlar lectura y escritura.

class Usuario {
  #edad;

  get edad() {
    return this.#edad;
  }

  set edad(nuevaEdad) {
    if (nuevaEdad < 0) {
      throw Error("La edad no puede ser menor que 0");
    }

    this.#edad = nuevaEdad;
  }
}
const u1 = new Usuario();

u1.edad = 30;

console.log(u1.edad);

El setter valida antes de guardar.

El getter controla cómo se devuelve el valor.

Desde fuera parece una propiedad normal.

Internamente existe lógica de control.

Visualización mental:

u1.edad = 30
      ↓
   setter
      ↓
 validación
      ↓
 almacenamiento

u1.edad
      ↓
   getter
      ↓
 devolución del valor

Los getters y setters permiten que el objeto proteja sus propias reglas.

Un objeto no solo almacena datos.

También puede controlar cómo se leen y modifican esos datos.

14. Herencia

La herencia permite crear una clase a partir de otra.

La nueva clase reutiliza propiedades y métodos ya existentes.

class Persona {
  constructor(nombre, apellido) {
    this.nombre = nombre;
    this.apellido = apellido;
  }

  saludar() {
    console.log(`Hola, soy ${this.nombre} ${this.apellido}`);
  }
}

class Alumno extends Persona {
  estudiar() {
    console.log("Estoy estudiando");
  }
}

const a1 = new Alumno("Juan", "García");

a1.saludar();
a1.estudiar();

Persona es la clase padre.

Alumno es la clase hija.

Alumno hereda las propiedades y métodos de Persona.

Además puede añadir métodos propios.

Resultado:

a1 puede usar:

saludar()   ← heredado
estudiar()  ← propio

Error típico: pensar que JavaScript copia todos los métodos del padre dentro del hijo.

Eso no ocurre.

15. ¿Qué hace realmente extends?

La palabra extends crea una relación de herencia entre dos clases.

class Alumno extends Persona {
}

Mucha gente imagina esto:

Persona ↓ copia ↓ Alumno

Eso es incorrecto.

Lo que realmente ocurre es una conexión entre prototipos.

Visualización mental:

a1
 ↑
Alumno.prototype
 ↑
Persona.prototype
 ↑
Object.prototype
 ↑
null

Cuando buscamos un método:

a1.saludar();

Paso 1:

a1

No lo encuentra.

Paso 2:

Alumno.prototype

Tampoco lo encuentra.

Paso 3:

Persona.prototype

Allí encuentra saludar().

La herencia funciona gracias a la cadena de prototipos.

extends no significa copiar.

extends significa conectar.

16. super()

Cuando una clase hija tiene su propio constructor, necesita ejecutar también el constructor del padre.

class Persona {
  constructor(nombre, apellido) {
    this.nombre = nombre;
    this.apellido = apellido;
  }
}

class Alumno extends Persona {
  #curso;

  constructor(nombre, apellido, curso) {
    super(nombre, apellido);

    this.#curso = curso;
  }
}

super() ejecuta el constructor de la clase padre.

Gracias a super() se inicializan las propiedades heredadas.

En este ejemplo, nombre y apellido se inicializan dentro de Persona.

curso se inicializa dentro de Alumno.

Visualización mental:

new Alumno(...)
        ↓
constructor Alumno
        ↓
super(...)
        ↓
constructor Persona
        ↓
nombre
apellido
        ↓
vuelve a Alumno
        ↓
curso

super() conecta la construcción del hijo con la construcción del padre.

Error muy importante:

constructor(nombre, apellido, curso) {
  this.#curso = curso;
  super(nombre, apellido);
}

En una clase hija no se puede usar this antes de llamar a super().

17. Sobrescritura de métodos (Override)

Una clase hija puede redefinir un método heredado.

class Persona {
  saludar() {
    console.log(`Hola, soy ${this.nombre} ${this.apellido}`);
  }
}

class Alumno extends Persona {
  saludar() {
    console.log(
      `Hola, soy ${this.nombre} ${this.apellido} y soy un alumno`
    );
  }
}

const a1 = new Alumno("Juan", "García");

a1.saludar();

Resultado:

Hola, soy Juan García y soy un alumno

Cuando JavaScript busca saludar():

Paso 1:

a1

No existe.

Paso 2:

Alumno.prototype

Lo encuentra aquí.

La búsqueda termina.

No necesita subir a Persona.prototype.

Sobrescribir no elimina el método del padre.

Simplemente el método más cercano tiene prioridad.

Es exactamente la misma idea que ya vimos con las propiedades heredadas.

Propiedad propia
      ↓
tiene prioridad
      ↓
sobre la heredada

Con los métodos ocurre lo mismo.

18. Resumen mental final de la clase

Las clases son una sintaxis más cómoda para trabajar con prototipos.

Las clases NO sustituyen a los prototipos.

new:

1. Crea un objeto
2. Lo conecta con su prototipo
3. Ejecuta el constructor
4. Devuelve el objeto

constructor() inicializa el objeto.

new es quien crea el objeto.

Todo lo que se asigna mediante this.algo = ... se convierte en una propiedad propia de cada instancia.

Los métodos definidos como saludar() {} viven en NombreClase.prototype.

Los métodos estáticos viven directamente en la clase.

Usuario
 └─ crearInvitado()

Las propiedades y métodos privados utilizan #.

Solo pueden utilizarse desde dentro de la clase.

Los getters controlan la lectura.

Los setters controlan la escritura.

Los setters permiten validar datos antes de guardarlos.

extends crea una relación de herencia.

La herencia funciona mediante prototipos.

Alumno.prototype
        ↑
Persona.prototype

super() ejecuta el constructor de la clase padre.

Permite inicializar las propiedades heredadas.

La sobrescritura da prioridad al método más cercano en la cadena de prototipos.

Si entiendes las clases pero no entiendes los prototipos, estás entendiendo solo la mitad del tema.

Si entiendes los prototipos, las clases dejan de parecer magia.
